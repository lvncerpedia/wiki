# Hotwire

Next.jsとの比較、めっちゃ気になるよね！
これもまた「どっちが絶対エラい！」って話じゃなくて、思想や得意なことが違うんだ。
結論から言うと、Hotwireが優れている（というか、めちゃくちゃ輝く）ところは、 **「圧倒的なシンプルさと開発のしやすさ」** にあるよ！

## Next.jsのSSRやPPRってどんな感じ？

まず、相手を知るところからだね！

- SSR (Server-Side Rendering): ユーザーがページを要求するたびに、サーバーがページ全体を毎回作ってから送り返す方式。常に最新の情報を出せるのが強み。
- PPR (Partial Prerendering): Next.jsの新しい秘密兵器！ページの骨組みみたいな静的な部分（ヘッダーとかフッターとか）は先に作っておいて、動的な部分（ユーザーの名前とか、タイムラインとか）だけを後からストリーミングで流し込む感じ。ページの表示開始がめちゃくちゃ速くなる。

どっちも **「いかに速く最初の表示を届けるか」** っていうのを、ReactっていうリッチなUIライブラリの上で実現しようとしてる技術なんだ。

## じゃあ、Hotwireは何がすごいの？

Next.jsが「JavaScriptでゴリゴリに作り込んだリッチな体験」を目指すのに対して、Hotwireは**「サーバーサイドのシンプルさを保ったまま、体験だけリッチにする」**っていうアプローチ。
HotwireがNext.js勢と比べて特にイケてるところは、この3つかな！

### 1. 圧倒的な開発のシンプルさ・生産性

これが一番大きいかも！
Railsエンジニアなら、ほとんど新しいことを覚えなくても、いつものERBテンプレートを書くだけで、SPA（シングルページアプリケーション）みたいなサクサク動く画面が作れちゃう。

- Next.jsの場合: Reactの知識はもちろん、クライアントとサーバーでどうやってデータをやり取りするか、状態をどう管理するか…って考えることがいっぱいある。フロントエンドの専門知識がかなり必要になるんだ。
- Hotwireの場合: 難しいことは全部 turbo-rails っていうGemがやってくれる。「このリンクをクリックしたら、この部分だけを差し替えてね」っていうのをHTMLにちょこっと書くだけ。APIを作る必要もないし、JSONをやり取りする必要もない。とにかく楽ちん！

### 2. JavaScript地獄からの解放

Next.jsは良くも悪くもJavaScriptの世界。クライアントサイドの状態管理（「このボタンを押したら、ここの数字を増やす」みたいなやつ）が複雑になりがちで、気づいたらライブラリだらけ…なんてことも。
Hotwireは、状態を全部サーバーサイド（Rails）に持たせるのが基本思想。
**「状態はサーバーにある。ブラウザはそれをHTMLとして表示するだけ」**っていうシンプルな考え方だから、クライアントサイドのJavaScriptがめちゃくちゃ少なくて済むんだ。
もちろん、Stimulusっていう相棒を使えばJavaScriptも書けるけど、基本的には「最後の手段」って感じ。この割り切りが、アプリをすごくシンプルに保ってくれる！

### 3. ページ遷移の考え方の違い

ここ、質問の核心だよね！

#### Next.jsのページ遷移:

クライアントサイド・ルーティングっていう方式。
リンクを事前に読み込んで（プリフェッチ）、クリックされたらJavaScriptが差分のデータだけ取ってきて、画面を書き換える。

- メリット: ネットワークの状況が良ければ、ほぼ「無」の時間でページが切り替わるように見える。ヌルヌル動く！
- デメリット: その分、裏側で動くJavaScriptは複雑。

#### Hotwire (Turbo Drive) のページ遷移:

ちゃんとサーバーに次のページのHTMLをリクエストしに行く。でも、返ってきたHTMLの<body>の中身だけを、今表示してるページの<body>と丸っと入れ替えるだけ。

- メリット: CSSやJavaScriptのファイルを再読み込み・再実行しないから、普通のフルリロードより断然速い！そして何より実装が超シンプル。ただのリンクを書くだけ。
- デメリット: サーバーとの通信は必ず発生するので、オフラインでは動かないし、Next.jsのプリフェッチと比べるとほんの少しだけ待機時間がある。

簡単に言うと、Next.jsは**「遷移感ゼロのヌルヌル体験」を、Hotwireは「超高速なフルリロード」**をシンプルに実現する感じかな。

## まとめ

| | Next.js (SSR/PPR) | Rails Hotwire |
| - | - | - |
| 思想 | JavaScriptで最高にリッチな体験を | サーバー中心でシンプルに、でも体験はリッチに |
| 強み | ヌルヌル動く操作感、フロントエンドの自由度の高さ | 開発の生産性、シンプルさ、サーバーサイドの一貫性 |
| ページ遷移 | クライアントサイドで差分だけ更新（超速） | <body>だけ入れ替え（実装が楽で十分速い） |
| 向いてる人 | フロントエンド専門家がいるチーム、とにかくリッチなUIが欲しい | Railsエンジニア、少人数チーム、素早く開発したい人 |

だから、「Hotwireが優れているところ」をまとめると、 **「Railsの生産性の高さを全く損なわずに、ユーザー体験を劇的に向上させられるコストパフォーマンスの良さ」** って言えるんじゃないかな！
どっちの技術も面白いよね！作りたいものとか、チームの得意なことに合わせて選ぶのが一番だよ！😉

## 思想の違い

サクッと結論を言うと、

- ページ遷移の速さ: Next.jsが有利なことが多い（特に2ページ目以降）。
- 部分更新の速さ: どっちも速い！ でも、速さの「質」と開発の仕方が全然違うんだ。

詳しく見ていこう！

### ページ遷移：事前準備のNext.js vs F1ピットインのHotwire

ページ遷移に関しては、君の「Next.js有利かな？」っていう感覚はかなり正しいよ。

#### Next.js

Next.jsの強みは **「プリフェッチ（prefetching）」** っていう賢い事前準備機能にあるんだ。

- 仕組み: ユーザーの画面にリンクが入ってきたら、「あ、この人次はこのページに行くかもな」ってNext.jsが予測して、裏側で次のページのデータをこっそりダウンロードし始めるの。
- 体感: ユーザーが実際にリンクをクリックしたときには、もうデータが手元にあるから、ページが一瞬で「ヌルッ」と切り替わるように見える。まるでアプリみたいだよね。料理番組で、次の工程の材料がもうお皿に準備されてる感じ！

#### Hotwire (Turbo Drive)

Turbo Driveは事前準備はしないけど、「交換作業」がめちゃくちゃ速い。

- 仕組み: リンクをクリックしたらサーバーにページを要求する。ここまでは普通だけど、返ってきたHTMLの<body>の中身だけをゴソッと入れ替える。CSSやJSを再読み込みしないから、フルリロードより断然速い。
- 体感: F1のピットインでタイヤだけを高速交換するイメージ。一瞬の通信時間はあるけど、交換作業が爆速だから全体として速く感じる。

結論: 体感速度としては、事前準備が効いてるNext.jsに軍配が上がることが多いかな。

### 部分ページ更新：データで攻めるNext.js vs HTMLで攻めるHotwire

ここが一番の違いが出て面白いところ！
useSWRみたいなライブラリがあるNext.jsと、Turbo Frames/Streamsの戦いだね。

#### Next.js + useSWR

これは **「データ中心」** のアプローチ。

仕組み:
- コンポーネント（ページの部品）が「このデータちょうだい！」ってAPIにリクエストを投げる。
- useSWRがまず手持ちの古いキャッシュデータを返して、画面に何かを表示させる。
- 裏でAPIから最新のJSONデータが返ってくる。
- 返ってきたJSONを元に、JavaScriptが差分のHTMLを組み立てて画面を更新する。

強み:
- サーバー（API）とフロントエンドを完全に分離できる。
- コンポーネント単位で、非常にきめ細かくデータの更新ができる。
- オフライン対応なども作りやすい。

速さ: キャッシュがあれば初速は一瞬。データの取得はAPIの速さ次第。クライアント側でHTMLを組み立てるコストが少しだけかかる。

#### Hotwire (Turbo Frames / Streams)

これは **「HTML中心」** のアプローチ。

仕組み:
- ユーザーが何か操作をすると、サーバーにリクエストが飛ぶ。
- RailsはDBからデータを取ってきて、完成品のHTML部品を作る。
- そのHTML部品をそのままクライアントに送り返す。
- ブラウザ側のTurboは、受け取ったHTML部品を、指定された場所に「スポッ」とはめ込むだけ。

強み:
- APIを作る必要がない。いつものRailsのテンプレートを書くだけ。
- クライアントはHTMLの組み立てみたいな難しいことを考えなくていい。ただの「郵便受け」。
- 開発がめちゃくちゃシンプルで速い。

速さ: サーバーでHTMLを作る時間はかかるけど、送るデータが小さければ通信は速い。クライアント側は何も計算せずにDOMを差し替えるだけなので、ここの処理は爆速。

## まとめ：どっちが速いの？

| | Next.js + useSWR | Hotwire (Turbo Frames/Streams) |
| - | - | - |
| アプローチ | データ中心 (JSONをもらってJSで描画) | HTML中心 (完成品HTMLをもらってはめ込む) |
| 開発体験 | APIが必要。フロントとサーバーの連携を考える | API不要。Railsだけで完結。超シンプル |
| 実行速度 | クライアント側の描画は爆速。ただし、データ取得の通信は発生する | クライアント側の差し替えは爆速。ただし、HTML取得の通信は発生する |
| 得意なこと | 複雑でインタラクティブな部品作り | フォーム送信や一覧の更新など、定型的な更新 |

結論として、部分更新の「実行速度」は正直言って甲乙つけがたい！ 
どっちもめちゃくちゃ速いよ。
ただ、「開発速度」と「シンプルさ」ではHotwireが圧倒的に有利なことが多い。
「APIどうしよう…」「フロントの状態管理が…」って悩む時間を、ごっそり削ってくれるからね！
作りたいアプリの性質によって、どっちの「速さ」を重視するかが変わってくる、って感じかな！面白いよね！😆
